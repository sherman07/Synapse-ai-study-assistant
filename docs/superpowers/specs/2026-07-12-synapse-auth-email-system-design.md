# Synapse Authentication Email System Design

## Goal

Make the public authentication experience feel like one high-quality Synapse product across login, sign-up, email verification, forgot password, and change password. All user-facing authentication emails must be branded and sent from Synapse, while Supabase remains the secure identity provider.

## User Experience

- Use one consistent Synapse authentication shell across all auth pages.
- Keep the current Synapse visual language, while improving hierarchy, spacing, responsive behavior, form feedback, and trust cues.
- Provide clear loading, success, failure, expired-link, and retry states.
- Show live password requirements, strength feedback, matching confirmation, show/hide controls, and accessible field-level errors.
- Avoid exposing Supabase wording in user-facing copy.
- Use generic recovery responses so users cannot discover whether an email address has an account.
- Ensure reset and verification links always land on the public Vercel pages in production.

## Email Architecture

The browser never sends auth emails directly and never receives a service-role key.

1. The Vercel frontend submits sign-up or password-reset requests to the Render FastAPI backend.
2. The backend validates the request and uses the Supabase service-role key server-side.
3. For sign-up, the backend creates or prepares the Supabase account without asking Supabase's default mailer to send the confirmation email, then generates a one-time confirmation action link.
4. For password reset, the backend generates a one-time recovery action link.
5. The backend sends a branded plain-text and HTML email through the configured SMTP provider.
6. The user follows the link to the Vercel verification or reset page.
7. The Supabase browser client consumes the one-time session and completes verification or password update.

## Service Responsibilities

### Vercel frontend

- Hosts the public auth pages and runtime config.
- Uses only the Supabase public URL and publishable/anon key.
- Calls `synapse-ai-backend` for sign-up and password-reset email requests.
- Never contains `SUPABASE_SERVICE_ROLE_KEY`, SMTP credentials, or other private secrets.

### Render `synapse-ai-backend`

- Hosts the FastAPI auth endpoints and all AI features.
- Generates Supabase action links using the service-role key.
- Sends Synapse-branded auth email through SMTP.
- Validates production redirect hosts and paths.
- Returns generic success responses for unknown recovery accounts.

### Render `synapse-data-api`

- Stores and reads application data.
- Keeps its existing Supabase and internal-token configuration.
- Does not send authentication email.

### Supabase

- Stores accounts, sessions, passwords, and verification state.
- Validates one-time confirmation and recovery tokens.
- Provides the browser session used to update the password.
- Allows only the production Vercel verification and reset redirect paths.

## Required Production Configuration

Supabase:

- Email/password authentication enabled.
- Email confirmation enabled for new accounts.
- Site URL set to the production Vercel URL.
- Redirect allowlist includes the production `verify.html` and `reset-password.html` URLs.

Render `synapse-ai-backend`:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SYNAPSE_FRONTEND_BASE_URL`
- `SYNAPSE_SMTP_HOST`
- `SYNAPSE_SMTP_PORT`
- `SYNAPSE_SMTP_SECURITY`
- `SYNAPSE_SMTP_USERNAME`
- `SYNAPSE_SMTP_PASSWORD`
- `SYNAPSE_SMTP_FROM_EMAIL`
- `SYNAPSE_SMTP_FROM_NAME=Synapse`

## Security and Failure Handling

- Service-role and SMTP secrets remain backend-only and are never committed.
- Redirect URLs are allowlisted by host and exact auth-page suffix.
- Recovery and confirmation tokens are generated by Supabase and are single-use.
- Email request responses do not reveal account existence.
- Provider failures return a safe user message and log only a masked email reference.
- The backend must fail closed when Supabase admin or SMTP configuration is missing.
- Existing accounts remain usable; the new flow applies to new confirmation and reset requests.

## Verification Plan

- Unit-test sign-up email-link generation, reset email-link generation, redirect validation, generic unknown-account behavior, SMTP message branding, and configuration failures.
- Run frontend auth regressions and static launch validation.
- Build the production frontend.
- Verify Render backend and data API health endpoints.
- Test production sign-up with a fresh email, confirm the email sender and public verification link, then log in.
- Test forgot password with a confirmed account, confirm the Synapse sender and public reset link, set a new password, and log in with it.
- Test expired or reused links and confirm the page gives a clear retry path.
